# ТЗ: Telegram-бот для расчёта реферальной комиссии по биржам

> Документ предназначен для реализации в Claude Code. Пиши по шагам (см. раздел «Этапы»), не пропускай раздел «Безопасность» — это ключевое требование заказчика.

---

## 1. Цель

Telegram-бот для одного пользователя (владельца). Пользователь:

1. Выбирает биржу из списка кнопок.
2. Вводит `uid` реферала.
3. Вводит диапазон дат (с — по).
4. Получает сумму реферальной комиссии за этот период по этому uid — для расчёта выплаты.

Бот обращается к официальным affiliate/rebate API бирж от имени аккаунта-партнёра (по API-ключам владельца), агрегирует начисления за период и возвращает итог с разбивкой по валютам/типам.

**Поддерживаемые биржи (7):** Bitget, Bybit, WEEX, Gate, OKX, KuCoin, MEXC.
Toobit и BitMart — НЕ реализуем (у Toobit нет API, у BitMart только broker-rebate, не affiliate).

---

## 2. Критически важная оговорка (заложить в продукт)

Данные из API — это **начисленная (accrued) комиссия**, а не всегда **рассчитанная к выплате (settled)** сумма. В частности Bybit прямо пишет в доке, что источник истины для расчёта выплат — Affiliate Portal, а не API. Поэтому:

- В ответе бота всегда выводить дисклеймер: *«Оценка начисленной комиссии по данным API. Для Bybit и при расхождениях сверяйтесь с веб-кабинетом партнёра.»*
- Не называть значение «к выплате» безусловно — называть «начислено за период по данным API».
- Хранить «сырые» записи (raw response) в логе запроса, чтобы можно было перепроверить расчёт.

## 3. Технологический стек

- Python 3.11+
- aiogram 3.x (Telegram)
- httpx (async HTTP-клиент, с таймаутами и проверкой TLS)
- pydantic v2 (валидация конфигов и моделей ответов)
- cryptography (Fernet — шифрование секретов at-rest)
- structlog или стандартный logging (структурные логи без секретов)
- Хранилище состояния FSM: memory (одному пользователю достаточно) или Redis (опционально)
- Деплой: Docker + docker-compose на VPS, запуск через long polling (без входящих портов)
- Менеджер зависимостей: poetry или uv; версии пинуются

### 4.1. Единый интерфейс адаптера

`bot/exchanges/base.py`:

```python
from abc import ABC, abstractmethod
from dataclasses import dataclass, field
from datetime import datetime
from decimal import Decimal

@dataclass
class CommissionLine:
    asset: str                  # валюта начисления, напр. "USDT"
    amount: Decimal             # сумма в этой валюте
    source: str | None = None   # SPOT / FUTURES / ... если биржа отдаёт

@dataclass
class CommissionResult:
    exchange: str
    uid: str
    date_from: datetime
    date_to: datetime
    lines: list[CommissionLine] = field(default_factory=list)  # разбивка по валютам
    total_usdt: Decimal | None = None     # если биржа даёт сразу в USDT
    raw_records_count: int = 0
    notes: list[str] = field(default_factory=list)  # предупреждения/оговорки
    settlement_note: str | None = None    # текст про accrued vs settled

class ExchangeAdapter(ABC):
    name: str
    # какой максимальный размер окна запроса поддерживает биржа (в днях)
    max_window_days: int = 30
    # поддерживает ли биржа фильтр по конкретному uid
    supports_uid_filter: bool = True

    @abstractmethod
    async def get_commission(self, uid: str, date_from: datetime, date_to: datetime) -> CommissionResult:
        ...
```

Реализация `get_commission` в каждом адаптере обязана:
1. Нарезать период на окна по `max_window_days` (utils/dates.py).
2. Для каждого окна сделать запрос, обработать пагинацию (если есть).
3. Просуммировать начисления по валютам через `Decimal` (никаких float для денег!).
4. Вернуть `CommissionResult` с разбивкой и оговорками.

## 5. Детальные спецификации по биржам

> ВАЖНО: эндпоинты и поля ниже верифицированы по официальной документации на момент составления ТЗ, но часть доков рендерится через JS. Перед финальной реализацией каждого адаптера обязательно открыть указанную ссылку и сверить: точный путь, версию API, имена полей ответа, параметры. Помечено «🔎 проверить вживую».

### 5.1. Bitget
- **Эндпоинт:** `GET /api/v2/broker/customer-commissions` («Get Agent Direct commissions»)
- **Док:** https://www.bitget.com/api-doc/affiliate/customerInfo/GetDirectCommissions
- **Параметры:** `uid`, `startTime`, `endTime` (мс), `idLessThan` (пагинация), `limit`, `coin`
- **Окно:** максимум 30 дней; по умолчанию возвращает вчерашний день (00:00–23:59 UTC+8), если не задать период
- **Аутентификация:** заголовки `ACCESS-KEY`, `ACCESS-SIGN`, `ACCESS-PASSPHRASE`, `ACCESS-TIMESTAMP`; подпись HMAC-SHA256 base64 по строке `timestamp + method + requestPath + body`
- **Ответ:** список начислений с суммой rebate и валютой; суммировать по валютам
- 🔎 проверить вживую имена полей суммы/валюты

### 5.2. Bybit
- **Эндпоинт (по uid):** `GET /v5/user/aff-customer-info` (param `uid`) — поля `commissions30Day`, `commissions365Day` (разбивка по монетам)
- **Эндпоинт (список):** `GET /v5/affiliate/aff-user-list` — поля `commissionsVol`, `commissions30Day`, `commissions365Day`
- **Док:** https://bybit-exchange.github.io/docs/v5/affiliate/affiliate-info и /affiliate-user-list
- **Аутентификация:** API-ключ с разрешением **ТОЛЬКО Affiliate**, только master UID. Заголовки `X-BAPI-API-KEY`, `X-BAPI-TIMESTAMP`, `X-BAPI-SIGN`, `X-BAPI-RECV-WINDOW`; подпись HMAC-SHA256
- **ОГРАНИЧЕНИЕ ПО ДАТАМ:** Bybit отдаёт скользящие окна `30Day`/`365Day`, а НЕ произвольный диапазон. Поэтому для Bybit:
  - Если запрошенный период ≈ последние 30 дней → вернуть `commissions30Day`.
  - Для произвольного диапазона точного API нет → бот должен честно вывести предупреждение: *«Bybit API отдаёт только скользящие 30/365 дней. Точная сумма за произвольный период — только в Affiliate Portal»*, и показать доступные 30/365-дневные значения как справочные.
  - В адаптере выставить `supports_uid_filter = True`, но обработать эту особенность отдельно (поле `notes`).
- **Дисклеймер:** обязательно — API не является источником истины для выплат (см. раздел 2)
- 🔎 проверить вживую

### 5.3. WEEX
- **Эндпоинт:** `GET .../rebate/affiliate/getAffiliateCommission` («Get Affiliate Commission»); вспомогательный `getChannelUserTradeAndAsset` (данные по рефералам + комиссия)
- **Док:** https://www.weex.com/api-doc/spot/rebate-endpoints/GetAffiliateCommission
- **Аутентификация:** заголовки `ACCESS-KEY`, `ACCESS-SIGN`, `ACCESS-PASSPHRASE`, `ACCESS-TIMESTAMP`
- 🔎 ОБЯЗАТЕЛЬНО проверить вживую: версию пути (`v1`/`v2`/`v3` — в источниках расходится), параметры (есть ли фильтр по uid и формат дат), имена полей ответа. Это самый «непроверенный» адаптер — заложить запас времени.

### 5.4. Gate (gate.io / gate.com)
- **Эндпоинт:** `GET /api/v4/rebate/partner/commission_history` («Partner obtains rebate records of recommended users»)
  - Дополнительно при необходимости: `/api/v4/rebate/agency/commission_history`, `/api/v4/rebate/broker/commission_history`
- **Док:** https://www.gate.com/docs/developers/apiv4/en/
- **Параметры:** `currency` (опц.), `user_id` (опц. — если задать, вернёт только этого реферала), `from`, `to` (unix-секунды), `limit`, `offset`
- **Окно:** максимум 30 дней; `from` по умолчанию = 7 дней назад
- **Ответ:** массив `PartnerCommission` — поля `commission_time`, `user_id`, `commission_amount` (string), `commission_asset`, `source` (SPOT/FUTURES). Это СПИСОК начислений — суммировать `commission_amount` по `commission_asset` самим (Decimal).
- **Аутентификация:** API key + secret, подпись Gate v4 (`KEY`, `SIGN`, `Timestamp`; SIGN = HMAC-SHA512 по `method\npath\nquery\nSHA512(body)\ntimestamp`)
- Пагинация через `limit`/`offset` — листать до конца.
- Проверено по доке — самый надёжный адаптер.

### 5.5. OKX
- **Эндпоинт:** `GET /api/v5/affiliate/invitee/detail` (param `uid`) — данные по приглашённому, включая накопленную комиссию
- **Док:** https://www.okx.com/docs-v5/en/#affiliate-rest-api
- **Аутентификация:** `OK-ACCESS-KEY`, `OK-ACCESS-SIGN`, `OK-ACCESS-TIMESTAMP`, `OK-ACCESS-PASSPHRASE`; SIGN = base64(HMAC-SHA256(`timestamp + method + requestPath + body`))
- 🔎 проверить вживую: точное имя поля с суммой комиссии (док JS-рендерится). Проверить, есть ли фильтр по диапазону дат или эндпоинт отдаёт накопленный итог — от этого зависит логика периода. Если диапазона нет — обработать как у Bybit (предупреждение + накопленное значение).

### 5.6. KuCoin
- **Эндпоинт:** `GET /api/v2/affiliate/queryMyCommission` («Get Commission»)
- **Док:** https://www.kucoin.com/docs-new/rest/affiliate/get-commission
- **Параметры:** `siteType`, `rebateType`, `rebateStartAt`, `rebateEndAt`, `page`, `pageSize`, `userId`, `dataType`
- **Ответ:** содержит поле `commission` (string) + `currency`, `takerVolume`, `makerVolume`. Суммировать по `currency`.
- **Аутентификация:** `KC-API-KEY`, `KC-API-SIGN`, `KC-API-TIMESTAMP`, `KC-API-PASSPHRASE`, `KC-API-KEY-VERSION: 2`; SIGN = base64(HMAC-SHA256(`timestamp + method + endpoint + body`)); passphrase тоже подписывается HMAC при key version 2
- Пагинация через `page`/`pageSize`.
- Проверено по доке (включая тело ответа) — надёжный адаптер.

### 5.7. MEXC
- **Эндпоинт:** `GET /api/v3/rebate/affiliate/commission` («Get Affiliate Commission Record, affiliate only»)
  - Детализация: `/api/v3/rebate/affiliate/commission/detail`; по рефералам: `/api/v3/rebate/affiliate/referral`
- **Док:** https://www.mexc.com/api-docs/spot-v3/rebate-endpoints
- **Параметры:** `startTime`, `endTime` (обязательны); возвращает per-invitee `spot`, `etf`, `futures`, `total` (USDT)
- **Аутентификация:** API key + secret, разрешение `SPOT_ACCOUNT_READ`; подпись HMAC-SHA256 по query string, заголовок `X-MEXC-APIKEY`
- **Ответ:** поле `total` (string) — сумма комиссии в USDT. Удобно: уже агрегировано в USDT.
- 🔎 проверить вживую: точный фильтр по uid (через `/referral` или параметр), формат пагинации.

## 6. Пользовательский сценарий (FSM)

Состояния aiogram FSM:

1. **idle** → команда `/start`. Проверка whitelist (middleware). Приветствие + кнопка «Рассчитать комиссию».
2. **choosing_exchange** → inline-клавиатура со списком бирж (только те, для которых сконфигурированы ключи — из `registry`). Callback с кодом биржи.
3. **entering_uid** → бот просит ввести uid. Валидация: непустой, соответствует ожидаемому формату (цифры/допустимые символы), длина в разумных пределах. Защита от инъекций (только whitelisted символы).
4. **entering_date_from** → ввод даты начала. Формат `YYYY-MM-DD`. Валидация. Можно дать inline-пресеты (вчера / 7 дней / 30 дней / свой диапазон).
5. **entering_date_to** → ввод даты конца. Валидация: `date_to >= date_from`, период не в будущем, разумный максимум (напр. ≤ 365 дней; предупредить, если биржа режет окно).
6. **confirm** → показать сводку запроса (биржа, uid, период) и кнопку «Подтвердить». Защита от случайных запросов.
7. **fetching** → показать «Считаю…», вызвать адаптер, обработать ошибки/таймауты.
8. **result** → вывести результат: разбивка по валютам, итог, кол-во записей, дисклеймер. Кнопки «Новый расчёт» / «Та же биржа, другой uid».

Везде — кнопка «Отмена» → возврат в idle и очистка состояния.

Пример вывода результата:
```
Биржа: Gate
UID: 1879043947
Период: 2026-05-01 — 2026-05-31

Начислено (по данным API):
  • 12.4567 USDT
  • 0.0031 BTC

Записей обработано: 52
⚠️ Оценка начисленной комиссии по данным API.
   Для выплаты сверьтесь с кабинетом партнёра.
```

## 7. Безопасность (ключевой раздел — реализовать полностью)

### 7.1. Контроль доступа к боту
- **Whitelist Telegram ID.** В конфиге `ALLOWED_TELEGRAM_IDS` (список int). Middleware на уровне всех апдейтов: если `from_user.id` не в списке — апдейт молча игнорируется (не отвечать, чтобы не палить существование бота), факт залогировать (id + время).
- Никаких «паролей в чате», никакого публичного доступа.
- Бот-токен — только из переменной окружения `BOT_TOKEN`, никогда в коде/гите.

### 7.2. API-ключи бирж — права доступа
- Для каждой биржи создавать ключ с **минимально необходимыми правами**: только affiliate/read. **Категорически НЕЛЬЗЯ** включать права на трейд и на вывод средств (withdraw). Прописать это явно в README как чек-лист.
- Для Bybit — разрешение **только Affiliate**.
- Где биржа поддерживает — привязать ключ к **IP-адресу VPS** (IP whitelist на стороне биржи). Это главный барьер при утечке ключа.

### 7.3. Хранение секретов (at-rest)
- Секреты НЕ в коде и НЕ в открытом виде в репозитории. `.gitignore` обязан включать `.env`, `*.key`, `secrets/`.
- Базовый вариант: все секреты в переменных окружения, инжектируются через docker secrets / systemd `EnvironmentFile` с правами файла `600`, владелец — сервисный пользователь.
- Усиленный вариант (реализовать): API-ключи бирж хранятся **зашифрованными** (Fernet, `cryptography`). Мастер-ключ шифрования (`MASTER_KEY`) подаётся только через переменную окружения в рантайме и нигде не сохраняется на диск. Модуль `security/secrets.py`:
  - `encrypt_secret(plaintext) -> token` (утилита для разовой подготовки зашифрованного конфига)
  - `decrypt_secret(token) -> plaintext` (в рантайме, ключи держатся в памяти процесса, не пишутся в файлы/логи)
- Запретить вывод секретов в логи/трейсбеки: при логировании ошибок маскировать ключи (`****`), не логировать заголовки запросов целиком.

### 7.4. Сетевое
- Все запросы к биржам — только HTTPS, с проверкой TLS-сертификата (httpx по умолчанию verify=True — не отключать).
- Таймауты на все запросы (connect + read, напр. 10/30 c). Ретраи с экспоненциальной задержкой на сетевые ошибки и 5xx (но НЕ на 4xx).
- Telegram — **long polling** (без входящих портов на VPS, меньше поверхность атаки). Если когда-то webhook — только HTTPS + `secret_token`.

### 7.5. Антифлуд и валидация ввода
- Rate-limit middleware: ограничить частоту запросов от пользователя (напр. не чаще 1 запроса в N секунд) — защита от случайного спама и от исчерпания лимитов API биржи.
- Жёсткая валидация всех вводов (uid — по whitelist символов; даты — строгий формат и границы). Никогда не подставлять пользовательский ввод в подпись/URL без санитизации.

### 7.6. Рантайм-изоляция
- Docker: образ на slim-базе, запуск под **non-root** пользователем, `read_only: true` для FS где возможно, `cap_drop: [ALL]`, без лишних volume.
- Не хранить персональные данные рефералов сверх необходимого. Логи запросов — с ретенцией и без чувствительных данных.
- Зависимости — пиновать версии; периодически прогонять `pip-audit`/`safety`.

### 7.7. Аудит
- Лог каждого запроса: кто (telegram id), какая биржа, какой uid, период, время, успех/ошибка, кол-во записей. Без секретов. Это нужно для разборов по выплатам.

## 9. Обработка ошибок

Адаптер и хэндлеры должны различать и по-человечески сообщать:
- Неверные/просроченные ключи или подпись (401/403) → «Проблема с доступом к API биржи X, проверьте ключи».
- Превышен лимит запросов (429) → ретрай с бэкоффом, затем «Биржа ограничивает частоту, попробуйте позже».
- Период больше окна биржи → автоматически нарезать, не показывать ошибку.
- Нет данных по uid → «По этому uid за период начислений не найдено».
- Сетевой таймаут → «Биржа не ответила вовремя».
- Любая необработанная ошибка → залогировать с трейсбеком (без секретов), пользователю — нейтральное сообщение.

Глобальный errors-хэндлер aiogram, чтобы бот не падал.

## 10. Тестирование

- Юнит-тесты на:
  - Подпись каждой биржи (сверить с примером из доки, где он есть).
  - Нарезку периода на 30-дневные окна (граничные случаи: ровно 30, 31, 60, 1 день).

  - Валидацию ввода (uid, даты).
  - Whitelist middleware (чужой id игнорируется).
- Мок-тесты адаптеров: httpx-mock на сохранённые примеры ответов (фикстуры из доков). Не дёргать живой API в CI.
- Опционально: ручной «smoke»-скрипт для проверки одного реального запроса по каждой бирже (запуск отдельно, не в CI, ключи из окружения).
- Финальная проверка: пройти полный сценарий в Telegram для как минимум Gate и KuCoin (самые надёжные спеки) живыми ключами на маленьком периоде.

## 13. Открытые вопросы / что обязательно проверить вживую перед сдачей

- **WEEX:** версия пути эндпоинта, наличие фильтра по uid, формат дат, имена полей ответа.
- **OKX:** имя поля суммы комиссии; есть ли фильтр по диапазону дат или только накопленный итог.
- **Bybit:** подтвердить, что произвольный диапазон недоступен; финализировать формулировку предупреждения.
- **MEXC:** как именно фильтровать по конкретному uid (через `/referral` или параметр), пагинация.
- **Bitget:** имена полей суммы/валюты, поведение пагинации `idLessThan`.
- Для всех: актуальные rate limits, чтобы корректно настроить бэкофф.
